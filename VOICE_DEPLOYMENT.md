# Public Internet Voice Deployment

The in-room voice feature uses WebRTC. Socket.io only forwards signaling messages; browser audio normally flows peer-to-peer.

For public internet games, peer-to-peer audio is not always possible because many players are behind NAT or firewalls. A TURN server fixes this by relaying audio when direct WebRTC cannot connect.

## Recommended Setup

1. Deploy the game server with HTTPS or behind a HTTPS reverse proxy.
2. Deploy a TURN server, for example `coturn`, on a public server.
3. Configure the backend with TURN credentials.
4. Restart the backend after changing `RTC_*` values.

## Server Env

Create `game-server/.env`:

```env
CLIENT_URL=https://your-game.example.com
RTC_STUN_URLS=stun:stun.l.google.com:19302
RTC_TURN_URLS=turn:turn.example.com:3478,turns:turn.example.com:5349
RTC_TURN_USERNAME=avalon
RTC_TURN_CREDENTIAL=change-this-password
RTC_FORCE_TURN=false
```

Use `RTC_FORCE_TURN=true` if you want maximum reliability and are okay with all voice traffic using TURN bandwidth.

The browser client calls `GET /api/voice/ice` after login to fetch this config. `game-client/.env.example` also includes optional client-side fallback values for development.

## Minimal coturn Example

Example `/etc/turnserver.conf`:

```conf
listening-port=3478
tls-listening-port=5349
fingerprint
lt-cred-mech
realm=turn.example.com
user=avalon:change-this-password
external-ip=YOUR_PUBLIC_SERVER_IP
no-multicast-peers
no-cli
```

Open firewall ports:

```text
TCP 3478
UDP 3478
TCP 5349
UDP 5349
UDP 49152-65535
```

For TLS TURN (`turns:`), configure valid certificates in coturn:

```conf
cert=/path/to/fullchain.pem
pkey=/path/to/privkey.pem
```

## Notes

- Chrome and most browsers require HTTPS for microphone access on public sites. `localhost` is the special development exception.
- TURN credentials sent to browsers can still be inspected by users. For a production app, short-lived TURN credentials generated by the backend are better than a single shared password.
- A 10-player mesh call creates many peer connections. It is fine for small Avalon rooms, but a future SFU service would scale better for larger voice rooms.
